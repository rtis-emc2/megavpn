# RTIS MegaVPN Roadmap 9.0.0

**Current release banner:** `7.1.1.0`
**Target release:** `9.0.0`
**Status:** architectural target / not implemented
**Current implementation baseline:** 8.x control plane, PostgreSQL source of
operational state, local generated artifact storage still present.
**Repository:** `github.com/rtis-emc2/megavpn`

Russian companion: [ROADMAP_9.0.0_RU.md](ROADMAP_9.0.0_RU.md).

## 1. Purpose

This roadmap defines the 9.0.0 architectural target for database-canonical
client artifacts and a cluster-ready Control Plane. It is a planning document,
not implementation evidence.

The central product decision is:

Generated client artifacts are not durable source-of-truth files. PostgreSQL
and the encrypted secret layer are the canonical source of truth for all
reproducible client configuration material. Downloadable files must be
reproducibly generated from canonical database state.

The same release target prepares the Control Plane for OCI containers, multiple
API and worker replicas, high availability and later cluster deployment.

## 2. Current Implementation Baseline

The current implementation baseline is the 8.x Control Plane and frontend work:
PostgreSQL stores operational state, while local generated artifact storage is
still present.

Observed current baseline:

- PostgreSQL stores clients, service accesses, instance revisions, jobs,
  resource locks, share links, subscription token hashes, audit events and
  encrypted secret references.
- Client generated artifacts are represented by `artifacts` rows containing
  `artifact_type`, `storage_path`, `content_hash`, `size_bytes` and status.
- The worker currently generates client artifacts and saves bytes through
  `SaveArtifactContent` under the configured artifact root, defaulting to
  `/var/lib/megavpn/artifacts`.
- Public share-link resolution loads a `share_links` row and then resolves a
  referenced `artifacts` row. The existing model therefore still depends on a
  stored path for serving generated artifact bytes.
- Runtime binary repository objects use `binary_artifacts.storage_path` and
  are separate from generated client configuration artifacts.
- `secret_refs` and the encryption service protect sensitive material, but the
  current generated artifact file itself is still a durable local file.
- Jobs and resource locks are database-backed and already provide a foundation
  for retry-safe workers, but full multi-worker cluster safety is not yet a
  release claim.
- Current deployment is systemd-oriented. API, worker, migration and agent
  binaries exist, and the API serves the web UI. OCI packaging and HA topology
  are not implemented by this roadmap step.

This baseline must not be described as stateless artifact generation. In the
current model, losing the local generated artifact directory can affect
downloads unless artifacts are rebuilt by an operator workflow.

## 3. Core Architectural Decision

For 9.0.0, PostgreSQL and the encrypted secret layer are the canonical source of
truth for all reproducible client configuration material.

Examples of reproducible generated material include:

- OpenVPN `.ovpn`;
- VLESS URLs;
- QR images;
- WireGuard client configuration;
- IPsec/IKE bundles;
- ZIP bundles;
- installation instructions;
- subscription payloads;
- downloadable client packages.

Policy:

1. Generated bytes are not canonical state.
2. A local file path must not be required to reconstruct an artifact.
3. Loss of the local artifact or cache directory must not cause permanent loss
   of client configuration material.
4. Any healthy API or worker replica with access to the same canonical database
   state and encryption keys must be able to generate the same artifact.
5. Artifact generation must be deterministic where the format permits it.
6. Generator version and source revision must be recorded.
7. Secret material must remain encrypted at rest.
8. Generated bytes must not be written into audit logs, job logs, traces or
   error payloads.
9. Local filesystem storage may exist only as temporary workspace, bounded
   cache, streaming buffer or compatibility layer during migration.
10. Local generated files must never be required for HA recovery.

## 4. Artifact Classes

### A. Reproducible Generated Artifacts

Examples:

- `.ovpn`;
- VLESS URL;
- QR image;
- generated ZIP bundle;
- WireGuard config;
- generated instruction bundle.

Canonical source:

- normalized database state;
- encrypted credentials;
- instance revision;
- endpoint data;
- generator version;
- generation policy.

Storage policy:

- generate on demand or asynchronously;
- stream directly when practical;
- optionally keep a bounded temporary cache;
- support safe regeneration after cache loss.

### B. Secret-Bearing Generated Artifacts

Examples:

- client private-key configuration;
- bootstrap environment;
- certificate bundle.

Canonical source:

- `secret_refs`;
- credential metadata;
- immutable generation inputs.

Additional controls:

- explicit RBAC;
- explicit audit;
- no-store response handling;
- one-time reveal only where the backend genuinely enforces it;
- short plaintext lifetime;
- no plaintext in logs, traces, audit payloads or cache metadata.

### C. Non-Reproducible Imported Binary Objects

Examples may include externally uploaded certificates or future opaque binary
content.

These are not ordinary generated artifacts. They may require a separate durable
storage decision such as encrypted PostgreSQL `bytea`, or external durable
object/blob storage referenced by PostgreSQL. Local application-node disk must
not be the durable backend. This roadmap does not choose the final binary-object
backend.

## 5. Target Logical Model

Detailed ERD and SQL migrations require a separate 9.0.0 design task. The target
logical model should include concepts equivalent to the following.

### Artifact Definition or Artifact Spec

- artifact type;
- client account;
- service access;
- source revision;
- generator type;
- generator version;
- safe generation parameters;
- status;
- `created_at`;
- `updated_at`.

### Artifact Generation

- definition or spec ID;
- generation attempt;
- source hash;
- output hash;
- size;
- status;
- `generated_at`;
- expiry or cache metadata;
- failure summary;
- worker identity.

### Artifact Delivery

- artifact definition or generation reference;
- channel;
- share link;
- download or delivery audit;
- safe destination hint;
- timestamps;
- status.

### Share Link

A share link must reference an artifact definition or bundle specification. It
must not depend on a permanent local pathname. A valid share link must be able
to regenerate the payload after application-node failover.

## 6. Target Generation Pipeline

The target generation path is:

1. Authorize the operator or public share-link request.
2. Load canonical database state.
3. Resolve encrypted secret material only when required.
4. Load the exact applied instance revision and endpoint data.
5. Validate all generation inputs.
6. Select a versioned generator.
7. Render bytes in an isolated temporary workspace or memory stream.
8. Calculate output hash and size.
9. Optionally persist bounded cache metadata.
10. Stream the response.
11. Audit the delivery without recording secret bytes.
12. Destroy or expire temporary material.

Two execution modes are required:

- synchronous generation for small payloads;
- asynchronous job-based generation for expensive bundles.

Generated cache must improve availability or performance only. It must not be a
prerequisite for download.

## 7. HA and Failover Implications

An application replica must be replaceable without moving generated client files
from the failed replica.

Failover requires access to:

- PostgreSQL canonical state;
- encrypted secret material;
- the external master key or secret-management service;
- compatible generator versions;
- optional shared cache, if configured.

Failover must not require replica-local generated artifact files.

Required recovery scenario:

1. Active API replica fails.
2. Another API replica receives the same request.
3. It reads the same canonical state.
4. It regenerates or retrieves a valid cached artifact.
5. Existing valid share links continue to work.
6. Output integrity remains verifiable.

Rejected HA models:

- bidirectional rsync of generated artifacts as the HA model;
- VM disk replication as the application-level source-of-truth strategy;
- shared local-path assumptions between replicas;
- direct rsync of a running PostgreSQL data directory.

## 8. OCI Containerization Target

Future OCI container candidates:

- Control Plane API;
- worker;
- frontend/static server or edge component;
- migration runner;
- administrative utility;
- scheduled maintenance jobs.

External state:

- PostgreSQL;
- encryption master key or KEK;
- TLS and signing secrets;
- durable imported-object storage if later required;
- external log and metric sinks.

Temporary container state:

- rendering workspace;
- bounded artifact cache;
- temporary downloads;
- build scratch directories.

The Node Agent remains a host-level system service for the 9.0.0 baseline
because it manages systemd services, host networking, packages, files, service
processes and firewall/runtime state. Containerizing the Node Agent is not part
of this roadmap unless a separate host-integration architecture is approved.

## 9. Cluster-Ready Control Plane

No mandatory orchestrator is selected in this roadmap. Kubernetes, Nomad,
Docker Swarm, Patroni, etcd, Consul, S3, MinIO and similar technologies require
separate architecture decisions.

### API

- stateless replicas;
- no local session authority;
- no local artifact authority;
- readiness and liveness probes;
- graceful shutdown;
- backward-compatible rolling deployment;
- shared authentication and session state.

### Worker

- multiple replicas only after concurrency safety is proven;
- database-backed jobs;
- resource locks;
- idempotent execution;
- retry-safe side effects;
- claim leases and expiry;
- no double application of destructive operations;
- singleton or leader-only maintenance tasks where required.

### Database

- external PostgreSQL;
- supported replication/failover design;
- fencing and split-brain prevention;
- backups and point-in-time recovery;
- schema migrations run as a controlled singleton job.

### Networking

- stable external endpoint;
- reverse proxy or load balancer;
- health-based routing;
- no node-specific public URL generation.

### Observability

- centralized logs;
- metrics;
- traces;
- replica identity;
- job ownership;
- failover events;
- artifact-generation metrics.

## 11. Acceptance Criteria

1. Deleting every local generated-artifact file does not prevent regeneration.
2. A clean API replica with only database and secret access can serve existing
   client downloads.
3. Two compatible replicas generate byte-identical output for deterministic
   artifact formats.
4. Output hashes are recorded and validated.
5. Existing valid share links survive API replica replacement.
6. A share link does not encode a replica-local path.
7. Restarting or replacing a worker does not permanently lose generation jobs.
8. Retried jobs do not create inconsistent duplicate state.
9. No plaintext secret is stored in images, logs, audit payloads or cache
   metadata.
10. A restored PostgreSQL database plus required external keys can regenerate
    previously available reproducible artifacts.
11. Cache loss is an availability or performance event, not a data-loss event.
12. Multiple API replicas can serve traffic concurrently.
13. Multiple workers can operate without conflicting mutations after the
    cluster-safety milestone is accepted.
14. Rolling deployment preserves supported API and job compatibility.
15. HA and backup/restore are tested separately.

## 12. Non-Goals for 8.0.0

- 8.0.0 does not implement the 9.0.0 artifact redesign.
- 8.0.0 may continue using current local artifact compatibility storage.
- 8.0.0 release closure must not be delayed by this roadmap document.
- No Docker or cluster migration is performed in the current branch task.
- No current artifact table or endpoint is changed.
- No current share-link behavior is changed.
- No orchestrator is selected.
- No production HA claim is made.

## 14. Documentation and Release Boundary

This roadmap does not change 8.0.0 scope, 8.0.0 remaining-debt accounting,
version metadata, migrations, endpoints, production code, CI workflows or
tracked frontend bundles. It records the 9.0.0 direction so future design tasks
can implement the architecture with explicit ADRs, tests and migration evidence.
