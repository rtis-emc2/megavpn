# Дорожная карта RTIS MegaVPN 9.0.0

**Текущий release banner:** `7.1.1.0`
**Целевой релиз:** `9.0.0`
**Статус:** архитектурная цель / не реализовано
**Текущий implementation baseline:** 8.x control plane, PostgreSQL source of
operational state, local generated artifact storage still present.
**Репозиторий:** `github.com/rtis-emc2/megavpn`

English companion: [ROADMAP_9.0.0.md](ROADMAP_9.0.0.md).

## 1. Назначение

Этот документ задает архитектурную цель 9.0.0: database-canonical клиентские
artifacts и Control Plane, готовый к контейнеризации, нескольким API/worker
репликам, high availability и последующему cluster deployment. Это planning
document, а не evidence реализации.

Ключевое продуктовое решение:

Generated client artifacts не являются durable source of truth. PostgreSQL и
encrypted secret layer являются каноническим источником истины для всего
воспроизводимого клиентского configuration material. Downloadable files должны
воспроизводимо генерироваться из канонического database state.

## 2. Текущий implementation baseline

Текущий baseline - это 8.x Control Plane и frontend work: PostgreSQL хранит
операционное состояние, но local generated artifact storage все еще
присутствует.

Фактический baseline:

- PostgreSQL хранит clients, service accesses, instance revisions, jobs,
  resource locks, share links, subscription token hashes, audit events и
  encrypted secret references.
- Client generated artifacts представлены строками `artifacts` с
  `artifact_type`, `storage_path`, `content_hash`, `size_bytes` и status.
- Worker сейчас генерирует client artifacts и сохраняет bytes через
  `SaveArtifactContent` под configured artifact root, по умолчанию
  `/var/lib/megavpn/artifacts`.
- Public share-link resolution загружает строку `share_links`, затем
  связанную строку `artifacts`. Поэтому текущая модель все еще зависит от
  stored path для выдачи generated artifact bytes.
- Runtime binary repository использует `binary_artifacts.storage_path` и
  отделен от generated client configuration artifacts.
- `secret_refs` и encryption service защищают sensitive material, но текущий
  generated artifact file остается durable local file.
- Jobs и resource locks уже database-backed и дают основу для retry-safe
  workers, но full multi-worker cluster safety еще не является release claim.
- Текущий deployment systemd-oriented. Есть API, worker, migration и agent
  binaries, API обслуживает web UI. OCI packaging и HA topology этим roadmap
  step не реализуются.

Этот baseline нельзя описывать как stateless artifact generation. В текущей
модели потеря local generated artifact directory может повлиять на downloads,
если artifacts не пересобраны операторским workflow.

## 3. Core architectural decision

Для 9.0.0 PostgreSQL и encrypted secret layer являются canonical source of truth
для всего reproducible client configuration material.

Примеры воспроизводимого generated material:

- OpenVPN `.ovpn`;
- VLESS URLs;
- QR images;
- WireGuard client configuration;
- IPsec/IKE bundles;
- ZIP bundles;
- installation instructions;
- subscription payloads;
- downloadable client packages.

Политика:

1. Generated bytes не являются canonical state.
2. Local file path не должен быть нужен для реконструкции artifact.
3. Потеря local artifact/cache directory не должна приводить к permanent loss
   client configuration material.
4. Любая healthy API или worker replica с доступом к той же canonical database
   state и encryption keys должна уметь сгенерировать тот же artifact.
5. Artifact generation должна быть deterministic там, где формат это допускает.
6. Generator version и source revision должны записываться.
7. Secret material должен оставаться encrypted at rest.
8. Generated bytes нельзя писать в audit logs, job logs, traces или error
   payloads.
9. Local filesystem storage может существовать только как temporary workspace,
   bounded cache, streaming buffer или compatibility layer на время миграции.
10. Local generated files не должны быть обязательны для HA recovery.

## 4. Классы artifacts

### A. Reproducible generated artifacts

Примеры:

- `.ovpn`;
- VLESS URL;
- QR image;
- generated ZIP bundle;
- WireGuard config;
- generated instruction bundle.

Canonical source:

- normalized database state;
- encrypted credentials;
- instance revision;
- endpoint data;
- generator version;
- generation policy.

Storage policy:

- generate on demand или asynchronously;
- stream directly, где это практично;
- optional bounded temporary cache;
- safe regeneration after cache loss.

### B. Secret-bearing generated artifacts

Примеры:

- client private-key configuration;
- bootstrap environment;
- certificate bundle.

Canonical source:

- `secret_refs`;
- credential metadata;
- immutable generation inputs.

Дополнительные controls:

- explicit RBAC;
- explicit audit;
- no-store response handling;
- one-time reveal только там, где backend действительно это enforce;
- короткий plaintext lifetime;
- отсутствие plaintext в logs, traces, audit payloads и cache metadata.

### C. Non-reproducible imported binary objects

Примеры могут включать externally uploaded certificates или future opaque binary
content.

Это не ordinary generated artifacts. Для них потребуется отдельное durable
storage decision: encrypted PostgreSQL `bytea` или external durable object/blob
storage, на который ссылается PostgreSQL. Local application-node disk не должен
быть durable backend. Этот roadmap не выбирает финальный backend для binary
objects.

## 5. Target logical model

Детальная ERD и SQL migrations требуют отдельной design task 9.0.0. Целевая
логическая модель должна включать концепции, эквивалентные следующим.

### Artifact Definition или Artifact Spec

- artifact type;
- client account;
- service access;
- source revision;
- generator type;
- generator version;
- safe generation parameters;
- status;
- `created_at`;
- `updated_at`.

### Artifact Generation

- definition/spec ID;
- generation attempt;
- source hash;
- output hash;
- size;
- status;
- `generated_at`;
- expiry/cache metadata;
- failure summary;
- worker identity.

### Artifact Delivery

- artifact definition/generation reference;
- channel;
- share link;
- download/delivery audit;
- safe destination hint;
- timestamps;
- status.

### Share Link

Share link должен ссылаться на artifact definition или bundle specification. Он
не должен зависеть от permanent local pathname. Действительный share link должен
уметь регенерировать payload после application-node failover.

## 6. Target generation pipeline

Целевой generation path:

1. Authorize operator или public share-link request.
2. Load canonical database state.
3. Resolve encrypted secret material only when required.
4. Load exact applied instance revision and endpoint data.
5. Validate all generation inputs.
6. Select a versioned generator.
7. Render bytes in isolated temporary workspace или memory stream.
8. Calculate output hash and size.
9. Optionally persist bounded cache metadata.
10. Stream the response.
11. Audit delivery без записи secret bytes.
12. Destroy или expire temporary material.

Нужны два execution mode:

- synchronous generation for small payloads;
- asynchronous job-based generation for expensive bundles.

Generated cache должен улучшать availability или performance, но не быть
обязательным условием download.

## 7. HA and failover implications

Application replica должна быть replaceable без переноса generated client files
с failed replica.

Для failover нужен доступ к:

- PostgreSQL canonical state;
- encrypted secret material;
- external master key или secret-management service;
- compatible generator versions;
- optional shared cache, если он настроен.

Failover не должен требовать replica-local generated artifact files.

Обязательный recovery scenario:

1. Active API replica fails.
2. Другая API replica получает тот же request.
3. Она читает тот же canonical state.
4. Она regenerates или retrieves valid cached artifact.
5. Existing valid share links continue to work.
6. Output integrity remains verifiable.

Отклоненные HA models:

- bidirectional rsync generated artifacts как HA model;
- VM disk replication как application-level source-of-truth strategy;
- shared local-path assumptions between replicas;
- direct rsync running PostgreSQL data directory.

## 8. OCI containerization target

Future OCI container candidates:

- Control Plane API;
- worker;
- frontend/static server или edge component;
- migration runner;
- administrative utility;
- scheduled maintenance jobs.

External state:

- PostgreSQL;
- encryption master key или KEK;
- TLS and signing secrets;
- durable imported-object storage, если позже понадобится;
- external log and metric sinks.

Temporary container state:

- rendering workspace;
- bounded artifact cache;
- temporary downloads;
- build scratch directories.

Node Agent остается host-level system service для baseline 9.0.0, потому что он
управляет systemd services, host networking, packages, files, service processes
и firewall/runtime state. Containerizing Node Agent не входит в этот roadmap,
пока отдельная host-integration architecture не будет утверждена.

## 9. Cluster-ready Control Plane

Этот roadmap не выбирает mandatory orchestrator. Kubernetes, Nomad, Docker
Swarm, Patroni, etcd, Consul, S3, MinIO и аналогичные технологии требуют
отдельных architecture decisions.

### API

- stateless replicas;
- no local session authority;
- no local artifact authority;
- readiness and liveness probes;
- graceful shutdown;
- backward-compatible rolling deployment;
- shared authentication and session state.

### Worker

- multiple replicas только после доказанной concurrency safety;
- database-backed jobs;
- resource locks;
- idempotent execution;
- retry-safe side effects;
- claim leases and expiry;
- no double application of destructive operations;
- singleton или leader-only maintenance tasks where required.

### Database

- external PostgreSQL;
- supported replication/failover design;
- fencing and split-brain prevention;
- backups and point-in-time recovery;
- schema migrations run as controlled singleton job.

### Networking

- stable external endpoint;
- reverse proxy или load balancer;
- health-based routing;
- no node-specific public URL generation.

### Observability

- centralized logs;
- metrics;
- traces;
- replica identity;
- job ownership;
- failover events;
- artifact-generation metrics.

## 11. Acceptance criteria

1. Удаление всех local generated-artifact files не мешает regeneration.
2. Clean API replica только с database and secret access может выдавать
   существующие client downloads.
3. Две compatible replicas генерируют byte-identical output для deterministic
   artifact formats.
4. Output hashes записываются и валидируются.
5. Existing valid share links survive API replica replacement.
6. Share link не кодирует replica-local path.
7. Restart или replacement worker не приводит к permanent loss generation jobs.
8. Retried jobs не создают inconsistent duplicate state.
9. Plaintext secret не хранится в images, logs, audit payloads или cache
   metadata.
10. Restored PostgreSQL database plus required external keys может
    regenerate previously available reproducible artifacts.
11. Cache loss является availability/performance event, а не data-loss event.
12. Multiple API replicas могут concurrently serve traffic.
13. Multiple workers могут работать без conflicting mutations после acceptance
    cluster-safety milestone.
14. Rolling deployment сохраняет supported API/job compatibility.
15. HA и backup/restore тестируются отдельно.

## 12. Non-goals for 8.0.0

- 8.0.0 не реализует artifact redesign 9.0.0.
- 8.0.0 может продолжать использовать current local artifact compatibility
  storage.
- 8.0.0 release closure не должен задерживаться этим roadmap document.
- Docker или cluster migration не выполняются в текущей branch task.
- Текущие artifact table или endpoint не меняются.
- Текущее share-link behavior не меняется.
- Orchestrator не выбирается.
- Production HA claim не делается.

## 13. Open architectural decisions

Отдельные ADR должны решить:

- exact artifact schema;
- generated-byte cache backend;
- policy for non-reproducible imported objects;
- synchronous versus asynchronous thresholds;
- generator version compatibility;
- OCI image split;
- cluster orchestrator;
- PostgreSQL HA implementation;
- distributed secret management;
- session-store model;
- leader election;
- worker lease semantics;
- cache invalidation;
- artifact retention;
- share-link regeneration policy;
- disaster-recovery RPO/RTO.

## 14. Documentation and release boundary

Этот roadmap не меняет 8.0.0 scope, 8.0.0 remaining-debt accounting, version
metadata, migrations, endpoints, production code, CI workflows или tracked
frontend bundles. Он фиксирует направление 9.0.0, чтобы будущие design tasks
реализовали архитектуру через явные ADR, tests и migration evidence.
