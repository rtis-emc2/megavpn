# Security and Release Review: 7.1.0.8

**Release:** `7.1.0.8`

## Scope

This review covers the traffic-accounting audit export filtering increment
after `7.1.0.7`:

- added Traffic Accounting UI controls for export `from`, `to`, `protocol`,
  `client_id`, `node_id` and `limit`;
- added recent-row preview filtering with the same selected values before CSV
  export;
- persisted export filter state in browser session storage;
- added regression tests for export filter request parsing, inverted date
  ranges and CSV header uniqueness;
- kept CSV export generation on the server-side retained dataset;
- updated traffic-accounting documentation, roadmap and release metadata.

## Security Posture

No new traffic data is collected. The platform still stores aggregate byte,
packet and flow counters plus attribution references and small collector
metadata. It does not store packet payloads, URLs, HTTP headers, DNS queries,
TLS session contents or per-destination browsing history.

The release does not introduce a new API endpoint or write path. The UI now
uses the existing read-only `/api/v1/traffic/accounting/export` filters, which
remain protected by the existing `traffic.read` permission. Server-side export
caps, retention cutoff enforcement, `Cache-Control: no-store` and attachment
disposition remain in place.

The filter preview is intentionally limited to rows already loaded in the UI.
The exported CSV is generated by the backend from retained rows matching the
selected filters, so operators do not rely on browser-side data as the audit
source of truth.

## Residual Risk

- CSV artifacts can contain client and node identifiers, so exported files must
  be handled as sensitive operational audit evidence.
- Browser session storage keeps the latest operator filter state for
  convenience. It contains filter values only, not exported traffic rows.
- Large installations still need live query-plan evidence under real
  cardinality.
