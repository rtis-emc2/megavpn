# Security and Release Review: 7.1.0.4

**Release:** `7.1.0.4`

## Scope

This review covers the traffic-accounting expansion after `7.1.0.3`:

- `megavpn-agent` now collects managed WireGuard peer byte counters through
  `wg show <interface> transfer`;
- `megavpn-agent` now parses managed OpenVPN status files and aggregates
  duplicate common names;
- managed OpenVPN server configs render `status-version 2`, a managed status
  file and a managed `ifconfig-pool-persist` path;
- managed WireGuard server configs render non-secret peer attribution comments;
- Control Plane traffic ingest can attribute WireGuard/OpenVPN samples to
  `service_accesses` through existing client metadata;
- default service-pack templates enable traffic accounting for managed
  OpenVPN/WireGuard components;
- migration `000015_vpn_traffic_accounting_defaults` updates existing default
  service-pack templates.

## Security Posture

The release keeps accounting at aggregate-counter scope. It does not store
packet payloads, URLs, headers, DNS names, TLS contents or per-destination
browsing history.

WireGuard collection uses public keys, interface names and byte counters only.
No WireGuard private keys or preshared keys are exported through accounting
metadata.

OpenVPN collection reads local status files generated by managed configs. Raw
operator-supplied OpenVPN configs are not modified automatically; if an operator
adds a custom status path, it remains node-local and is only parsed by the
agent.

Server-side ingest still validates node ownership for reported instances,
rejects malformed UUIDs and negative/empty counters, and resolves
`service_access` attribution inside the Control Plane. Agent submissions remain
node-scoped and signed.

## Residual Risk

- Existing OpenVPN/WireGuard instances must be re-applied after upgrade so their
  rendered configs include the managed status path or attribution comments.
- Agent counter baselines are in memory. After agent restart, the first
  interval is baseline-only.
- WireGuard/OpenVPN accounting currently provides byte aggregates, not
  destination-level flow logs. Long-term audit export and partitioned retention
  still need live-volume validation.
- Live-node validation must cover service restart, reconnect and counter reset
  scenarios before broad production collection is enabled.
