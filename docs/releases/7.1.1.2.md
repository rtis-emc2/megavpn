# Release 7.1.1.2

**Release:** `7.1.1.2`

Release `7.1.1.2` prioritizes external provider egress and initial console
responsiveness. It extends the existing selected-client gateway model without
changing node-wide default routing.

Migration `000025_external_egress_runtime_guards` expands the database runtime
allowlist and enforces the L2TP client/server UDP/1701 reservation at the
PostgreSQL boundary. Both reservation paths lock the node row, so concurrent
requests cannot place an external L2TP/IPsec client and a managed XL2TPD server
on the same host.

## Scope

- Adds runtime-ready L2TP over IPsec external provider gateways using
  strongSwan, `xl2tpd` and `pppd`.
- Adds runtime-ready VLESS and Shadowsocks external provider gateways using a
  dedicated loopback-only Xray proxy process.
- Keeps external-provider routing scoped to explicitly assigned global client
  access groups. Unselected clients and node processes keep their existing
  routing policy.
- Loads readiness endpoints in parallel, then prioritizes the five authenticated
  dashboard/node/instance/client/job requests needed for the first populated
  render. Secondary inventory starts in parallel after that render so it cannot
  starve the initial view. Each core API request is capped at 12 seconds.
- Adds strict import validation, protocol-specific UI fields, runtime probe and
  cleanup behavior, operator documentation and regression tests.

## Runtime Model

| Provider protocol | Managed runtime | Selected traffic path |
| --- | --- | --- |
| OpenVPN | OpenVPN client interface | Xray mark -> dedicated table -> provider interface |
| WireGuard | `wg-quick` client interface | Xray mark -> dedicated table -> provider interface |
| L2TP/IPsec | strongSwan + `xl2tpd` + managed PPP interface | Xray mark -> dedicated table -> provider interface |
| VLESS | Dedicated Xray client process | Group outbound -> `127.0.0.1` SOCKS -> provider |
| Shadowsocks | Dedicated Xray client process | Group outbound -> `127.0.0.1` SOCKS -> provider |

L3 provider routes are never installed in the node main table. VLESS and
Shadowsocks listeners bind only to loopback. A missing runtime path fails the
selected group closed instead of silently using normal node egress.

## Security And Compatibility

- Plaintext VLESS and `allowInsecure` are rejected; TLS or REALITY is required.
- VLESS and Shadowsocks imports reject unknown fields, control characters,
  unsupported ciphers/transports and plugin execution.
- L2TP/IPsec imports use a strict key allowlist and require PSK plus
  username/password. Unencrypted L2TP remains blocked.
- Only one active external L2TP/IPsec client deployment is allowed per node due
  to node-scoped UDP/1701 and `xl2tpd` ownership. A managed XL2TPD server and an
  external L2TP/IPsec client cannot share a node; apply also checks for an
  unmanaged UDP/1701 listener before changing runtime files.
- Provider secrets remain encrypted at rest and are hydrated only into a
  claimed apply job for the owning node.

## Validation

The maintained release gates are:

```bash
make build
go test ./...
go test -race ./...
go vet ./...
scripts/ci/docs-consistency.sh
scripts/ci/self-test.sh
scripts/ci/release-gate.sh
```

Live provider authentication, MTU, DNS, throughput, systemd and Linux policy
routing still require disposable-node evidence before production promotion.
L2TP/IPsec certificate authentication and provider IPv6 routes are outside this
release scope.
