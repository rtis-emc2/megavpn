# Release 8.0.0

Release `8.0.0` introduces the MegaVPN Console frontend architecture.

Current status: release candidate groundwork with VLESS client access groups,
Firewall preview/apply/disable, Clients core/artifacts, Clients delivery,
client routes/access maintenance/config cleanup, existing Instances runtime
control, service pack instance creation, manual instance creation, service pack
management and runtime artifact list/import, Nodes create/edit,
observability/diagnostics/inventory, secure SSH access-method creation,
Nodes bootstrap/security/control workflows and guided Agent
Registration/Onboarding, Certificates/PKI, Platform Settings, Mail/Delivery
settings and Platform Access users/invites/sessions, Backhaul
list/detail/actions and Route Policy preview/apply/cleanup migrated to the new
console where backend endpoints exist. Broader write-workflow parity is still
tracked per domain.
Backend binary/version metadata remains `7.1.1.0` until the separate version
synchronization task is completed.
Final cutover remains NO-GO until version metadata is synchronized, a full
release gate passes without workstation skips, live disposable API/DB/node
smoke is complete, responsive workflow evidence is captured and release owners
sign off the remaining backend-missing/future-scope debt.

## Highlights

- New React + TypeScript + Vite frontend workspace under `frontend/`.
- New MegaVPN UI component kit and CSS design tokens.
- Russian and English i18n resources with key parity checking.
- Typed API client preserving cookie session and CSRF behavior.
- TanStack Query server-state layer.
- Legacy static UI preserved under `/legacy/`.
- Go static serving updated for `/legacy/`, Vite-built root assets and direct
  frontend deep links.
- CI/release-gate frontend checks expanded to typecheck, lint, test, i18n and build.
- GitHub Actions workflow pins are kept as commit SHAs and updated to upstream
  node24 runtime action versions for checkout, setup-go, setup-node and
  upload-artifact.
- Text LF guards protect `.gitattributes`, CI workflow files, documentation
  guard scripts and FE8 evidence docs from collapsed raw GitHub formatting.
- Frontend serving smoke validates root UI, deep links, legacy UI, API
  non-shadowing and static asset 404 behavior.
- Frontend static guards block raw page-level API calls, auth token browser
  storage, unreviewed HTML sinks and production console logging.
- `Clients -> Groups -> VLESS` is wired to typed generic client access group
  endpoints for create/edit, member preview/apply/remove, scope update and sync
  preview/apply.
- `Clients` core workspace is wired for list/detail, create, activate/suspend,
  revoke/delete, single-client VLESS access assignment and client artifact
  list/build/download/delete.
- Client delivery workflows are wired for artifact share links, VLESS
  subscriptions and email delivery with transient one-time secret display.
- Client maintenance workflows are wired for generic client edit, route
  list/create/update/delete, service access list/rotation/revoke/delete and
  generated config cleanup.
- Existing `Instances` runtime control is wired for list/detail, runtime state,
  revisions/rollback, apply/reapply, lifecycle actions, diagnostics,
  delete/force-delete and job tracking.
- `Services -> Service Packs` is wired for list/detail, JSON create/update,
  enable/disable/delete and create instance from pack.
- `Instances` manual create and spec replace are wired to backend instance
  endpoints with explicit confirmation where required.
- `Services -> Runtime Artifacts` is wired for runtime binary artifact list,
  safe metadata view and URL import.
- `Nodes` is wired for node list/detail, control-plane node create/edit safe
  profile metadata, agent/runtime state, maintenance mode, inventory view/sync,
  capability install/verify,
  diagnostics retry/run, service discovery list/import, enrollment tokens,
  secure SSH access-method creation, bootstrap/reinstall job queueing, host-key
  scan/pin, SSH session ticket launch, agent token rotation, retire/force-retire
  and async job tracking.
- `Nodes -> Security` secure SSH access-method creation uses explicit host-key
  selection and independent verification, one-way private-key submission,
  atomic encrypted backend persistence, redacted result handling and
  non-skipping PostgreSQL integration evidence.
- `Nodes -> Bootstrap` manual bootstrap bundle reveal/download is wired without
  `/legacy/` for existing completed manual bundle runs: availability is exposed
  through `manual_bundle_available`, reveal/download require sensitive-material
  acknowledgement, responses use no-store handling, downloads preserve exact
  bytes, secret references are not exposed and reveal/download actions are
  audited. Evidence includes the real PostgreSQL store test, real
  HTTP/router/PostgreSQL test and GitHub Actions run `29391281058` with
  required bundle tests executed without skips.
- `Nodes -> Onboarding` is wired without `/legacy/` for backend-derived
  onboarding milestone status, enrollment-token issue/reissue, explicit
  one-time token handling, response-loss recovery guidance, SSH/manual-bundle
  bootstrap selection, bootstrap job submission/tracking, registration waiting,
  first-heartbeat waiting, guided inventory synchronization, asynchronous
  inventory-job tracking and backend-derived ready-state derivation.
  Permission checks stay split between `node.bootstrap` and `node.write`, the
  browser does not call `/agent/*`, and accepted jobs are not treated as
  completed registration, heartbeat or inventory milestones. Evidence includes
  real disposable HTTP/PostgreSQL registration, signed heartbeat, signed
  inventory, job polling/result submission, replay protection, diagnostics
  `inventory_ok`, GitHub Actions run `29401792602` for protocol evidence and
  run `29415883087` for final functional UI evidence.
- `Platform -> Certificates` is wired for certificate list/detail, import with
  backend preview, self-signed certificate creation, managed CA creation,
  issue-from-CA, set default, revoke, CA cascade delete and managed service PKI
  root creation. Private key material remains transient form state and is not
  logged, stored or rendered after submit.
- `Platform -> Settings`, `Platform -> Mail / Delivery` and `Platform -> Access
  / RBAC` are wired for control-plane TLS read/save/apply, runtime preflight,
  mail settings read/save/test, users list/detail, invite list/create and
  session list/revoke. SMTP passwords and invite/session secrets are not logged,
  persisted or rendered.
- `Infrastructure -> Backhaul` is wired for link list/detail, apply, probe,
  transport promote and route projection enable/disable with confirmation and
  backend job tracking. Transport config payloads and secret refs are not
  rendered.
- `Network Policy -> Route Policy` is wired for node-scoped list/detail,
  backend preview, apply and cleanup. Apply requires a fresh successful preview
  for the selected node; stale preview disables Apply.

## Operator Workflow Impact

- Root `/` opens the new 8.0.0 console.
- `/legacy/` opens the previous static UI during migration.
- `Clients -> Groups` is the source of truth for access groups; the VLESS path
  is usable in the new console without `/legacy/`.
- `Clients` core operations are usable in the new console without `/legacy/`
  for client list/detail/create/status/delete/revoke, single-client VLESS
  preview/apply/remove and artifacts list/build/download/delete.
- `Clients -> Delivery` supports share link create/rotate/revoke, VLESS
  subscription create-or-rotate/revoke and email delivery in the new console
  without `/legacy/`.
- `Clients -> Routes/Maintenance` supports route list/create/delete, service
  access list/rotation/delete and generated config cleanup in the new console
  without `/legacy/` where the backend supports the operation.
- `Instances` runtime control works in the new console without `/legacy/` for
  existing service instances. Access group materialization is read-only and
  links operators to `Clients -> Groups`.
- `Services -> Service Packs` works in the new console without `/legacy/` for
  service pack list/detail/create/update/enable/disable/delete and create
  instance from pack.
- `Instances -> Manual instance` works in the new console without `/legacy/`
  where the backend accepts manual instance creation.
- `Instances -> Spec` replacement works in the new console without `/legacy/`
  through the backend spec replace endpoint and explicit confirmation.
- `Services -> Runtime Artifacts` works in the new console without `/legacy/`
  for list, metadata view and URL import.
- `Instances` no longer owns primary VLESS group management in the new information architecture.
- Firewall address groups, policies, rules, node preview, apply, node state and
  emergency disable are connected in the new UI without `/legacy/`.
- Firewall UI keeps preview-before-apply semantics.
- `Nodes` observability, diagnostics, inventory, capabilities and service
  discovery workflows for existing nodes are usable in the new console without
  `/legacy/`.
- `Nodes` control-plane create/edit safe profile metadata works in the new
  console without `/legacy/` through `POST /api/v1/nodes` and
  `PUT /api/v1/nodes/{id}`. This does not create SSH access secrets, perform
  agent registration or claim enrollment/online state before the backend
  reports it.
- `Nodes` bootstrap/security/control workflows are usable in the new console
  without `/legacy/` for configured nodes: enrollment token create/rotate/revoke,
  bootstrap/reinstall job queueing, host-key scan/pin, SSH session ticket launch,
  agent token rotation and retire/force-retire.
- `Nodes -> Security` supports secure SSH access-method creation without
  `/legacy/`: operators enter host/port/user, scan and explicitly select a host
  key, confirm independent verification, submit the private key once to
  encrypted backend storage and receive a redacted result. This configures and
  persists the method; it does not prove live SSH connectivity, bootstrap or
  agent enrollment.
- `Nodes -> Bootstrap` supports secure manual bootstrap bundle reveal/download
  without `/legacy/` for already completed manual bundle runs: operators use
  `manual_bundle_available`, explicitly acknowledge sensitive material, reveal
  into transient UI state or download the backend-generated no-store `.env`
  attachment, and every reveal/download is audited. This does not prove bundle
  execution, agent installation, registration or heartbeat.
- `Nodes -> Onboarding` supports the guided Agent Registration/Onboarding
  sequence in the new console without `/legacy/`: create or select a node,
  issue or reissue an enrollment token when recommended, select SSH bootstrap
  or manual-bundle bootstrap, submit bootstrap, wait for `registered_at`, wait
  for first heartbeat, queue guided inventory sync, wait for actual inventory
  evidence and reach backend-derived ready state only after registration,
  heartbeat, inventory and healthy communication state are present.
- `Platform -> Certificates` works in the new console without `/legacy/` for
  certificate list/detail, import preview/apply, self-signed create, managed CA
  create, issue from CA, set default, revoke/delete and managed service PKI root
  creation where the backend supports the exact operation.
- `Platform -> Settings` works in the new console without `/legacy/` for runtime
  preflight, control-plane TLS read/save and TLS apply with confirmation and job
  tracking.
- `Platform -> Mail / Delivery` works in the new console without `/legacy/` for
  SMTP/mail settings read/save and mail test. SMTP password input is masked and
  write-only.
- `Platform -> Access / RBAC` works in the new console without `/legacy/` for
  users list/detail, invite list/create and sessions list/revoke with
  permission-aware disabled states and confirmation for session revoke.
- `Infrastructure -> Backhaul` works in the new console without `/legacy/` for
  existing backhaul link list/detail, apply, probe, transport promote and route
  projection enable/disable where the backend supports the operation.
- `Network Policy -> Route Policy` works in the new console without `/legacy/`
  for node-scoped route policy preview/apply/cleanup where the backend supports
  the operation.
- Unsupported write workflows are disabled or linked to `/legacy/`; no
  fake-success actions are allowed.

## Known Limitations

- The first architecture pass implements the new shell, typed data loading and read-path pages.
- High-risk write workflows remain disabled or legacy-only unless the backend
  action is explicitly wired.
- Client access group preview/apply is fully connected for VLESS only.
  Non-VLESS access services remain catalog-only until backend materialization
  is implemented.
- Rule reorder remains unavailable because the backend has no reorder endpoint.
- Live/staging operator validation is still required before production use of
  strict firewall apply.
- Nodes control-plane create/edit safe profile metadata, secure SSH
  access-method creation, manual bootstrap bundle reveal/download and guided
  Agent Registration/Onboarding are migrated.
- Nodes agent identity revoke, reboot, emergency cleanup and stale rotation
  cleanup remain legacy-only or future scope.
- Nodes service discovery ignore/unignore remains legacy-only or future scope.
- Live external-node onboarding smoke and live/staging validation are still
  required before production reliance on strict node onboarding workflows.
- Nodes SSH session launch is connected through a backend-issued short-lived
  terminal URL; the new UI does not implement browser-side SSH and does not
  persist SSH credentials or terminal tickets.
- Node runtime reconcile is connected through the backend diagnostics endpoint;
  it may queue dependent runtime jobs as defined by backend logic and still
  requires live/staging operator validation before production reliance.
- Runtime artifact delete remains unavailable because the backend has no binary
  runtime artifact delete endpoint in this release.
- Service pack create/update has no separate validation endpoint; backend
  validation runs during the real create/update mutation.
- Instance spec replace has no separate preview endpoint or draft-save HTTP
  route in this release.
- Client core provisioning through single-client VLESS access assignment,
  generic client edit, client artifacts, share links, VLESS subscriptions,
  email delivery/history, route list/create/update/delete, service access
  rotation/revoke/delete and generated config cleanup are connected in the new
  console.
- Client access rotation/config cleanup have no backend preview endpoints; the
  new UI requires explicit confirmation and then calls the real backend
  mutation with backend validation and job/result tracking.
- Client delivery history is connected through the backend client-scoped
  history endpoint with masked destination hints and redacted safe error
  summaries.
- Client email delivery is real and synchronous, but the backend sends the
  client's available artifacts/configs and does not accept an artifact-specific
  email payload yet.
- Platform invite revoke remains unavailable because the backend has no browser
  invite revoke endpoint in this release.
- Direct Platform user lifecycle mutations, including user status changes,
  reset-password, resend-invite and delete, remain legacy-only/future scope.
- Certificate import requires backend preview before apply; there is no separate
  browser detail endpoint, so detail is derived from the real certificate list.
  Private key material is submitted only to backend mutation endpoints and
  cleared from local form state on close/success.
- Backhaul create/delete remain legacy-only/future scope in the new console.
  Backhaul apply/probe/promote/route projection are connected for existing
  links and still require live/staging operator validation before production
  reliance.
- Route Policy preview/apply/cleanup is connected as a node-scoped workflow;
  there is no separate route-policy list endpoint, so the new UI derives
  list/detail from managed nodes.
- Full workflow parity should be completed page by page against
  `docs/FRONTEND_AUDIT_8.0.0.md`,
  `docs/FRONTEND_ENDPOINT_PARITY_8.0.0.md` and
  `docs/FRONTEND_WRITE_WORKFLOWS_8.0.0.md`.
- Backend binary version synchronization to `8.0.0` is still required before
  final release readiness.
- Final 8.0.0 cutover remains NO-GO until the remaining legacy-only workflows,
  live/staging operator validation and version synchronization are complete.
