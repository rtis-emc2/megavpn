# Release 7.1.1.0

Release `7.1.1.0` is the current stabilization baseline for the MegaVPN
Control Plane, agent runtime and operator console.

## Summary

`7.1.1.0` is the merge-ready hardening release for the VLESS group, managed
firewall and release-gate stabilization line. It keeps the idempotent
service-pack rollout behavior from `7.1.0.30` and adds safer firewall disable
lifecycle storage, explicit VLESS group membership management, stronger CI
release gates and migration invariants.

The release keeps the `7.1.0.26` durable VLESS identity model: VLESS identity
is stored at the client level, replacement ingress nodes inherit the already
issued client UUID during provisioning, and VLESS group membership can be
managed independently from a single runtime instance.

## Stabilization Scope

- Clean install and upgrade path on a new Ubuntu host.
- PostgreSQL migrations on disposable databases.
- Signed agent channel with replay-window checks.
- Typed privileged job APIs and job-type permission matrix.
- Node bootstrap, update and emergency cleanup workflows.
- Service-pack and manual instance creation, apply and runtime convergence.
- Centralized VLESS access groups for client routing policy.
- Managed backhaul UX with one active ingress-to-egress transport and explicit
  standby promotion.
- Traffic-camouflage Nginx/Xray ingress with explicit fallback website and
  managed rollback on failed validation/apply.
- Managed firewall catalog with protocol presets and controlled default-policy
  enforcement for nftables apply.
- Traffic accounting with signed agent ingest API, PostgreSQL aggregate
  storage, 180-day retention cleanup, `traffic.read` RBAC, operator UI,
  managed Xray/WireGuard/OpenVPN byte-counter collectors and no-store CSV
  export.
- Route-policy enforcement for ingress client traffic and local Xray/VLESS
  system egress through managed nftables chains and managed backhaul tables.
- OpenVPN full-tunnel defaults with managed forwarding and NAT policy.
- OpenVPN, WireGuard, Xray/VLESS, Shadowsocks, Nginx and Backhaul smoke matrix.

## Notable Changes

### Service Packs

- Service-pack creation is selective: operators can create only chosen
  components, override per-component listen ports and choose OpenVPN CA
  material without installing every service in a template.
- Service-pack creation has an explicit completed state: the submitted form is
  preserved and locked after success, while direct actions open created
  instances or start a separate new rollout.
- Semantic service-pack deduplication is implemented in API/UI with database
  repair for historical duplicate default pack rows.

### VLESS And Xray

- Client access groups now have a generic service-catalog model exposed through
  `GET /api/v1/client-access-services`. VLESS is the only active materialized
  service in this release; OpenVPN, WireGuard, L2TP/IPsec, HTTP Proxy,
  Shadowsocks, MTProto and SOCKS are visible as coming-soon/planned catalog
  entries and fail closed for group materialization.
- `Instances` no longer exposes VLESS groups as a primary operator workspace.
  `Clients -> Groups` is the source of truth for group policy and membership;
  instances are runtime materialization targets only.
- VLESS client provisioning syncs active access-group catalog entries into the
  selected Xray instance before validating the chosen group.
- Operators manage VLESS client membership globally in `Clients -> Groups`;
  instance screens show runtime materialization and link back to the source of
  truth.
- The client access group members workflow supports paginated server-side
  search, visible-page selection, all-filtered selection, pasted client refs,
  mandatory dry-run preview and one bounded apply fanout per affected instance.
- VLESS group membership, policy, scope and status mutations queue affected
  Xray apply jobs so runtime client access follows catalog changes without
  manually rebuilding every client.
- Selected-egress groups are materialized into concrete Xray outbound and
  source-route metadata.
- VLESS client identity is stable across Xray/VLESS ingress instances and node
  replacement.
- Xray UUID rotation preserves the client's selected VLESS access group instead
  of falling back to a stale implicit `route` value.
- Stale implicit group metadata falls back to an active catalog group, while
  explicit invalid operator choices still fail closed.
- Client provisioning action rows use compact scoped buttons instead of
  inherited full-width modal bars.

### Nginx Camouflage

- Nginx capability install is resilient for camouflage ingress, with nginx.org
  to Ubuntu repository fallback on repository/package-stage failures.
- Ubuntu Nginx fallback preserves existing packages until a distro candidate is
  confirmed and reports precise apt failure commands.
- Generated TLS-enabled Nginx edge configs include an HTTP listener that
  redirects plain HTTP traffic to HTTPS before camouflage/fallback routing.
- Nginx instance and emergency cleanup reload shared Nginx when managed configs
  remain and stop it when all MegaVPN-managed edge configs are gone.
- Nginx instance apply can recover a missing binary through the managed
  nginx.org-to-Ubuntu fallback installer.

### Backhaul And Route Policy

- Route-policy preview lets operators inspect projected client routes,
  VLESS/Xray system egress routes, blocked/observe-only reasons and managed
  nft/ip-rule primitives before queueing `node.route_policy.apply`.
- Agent job results capture systemd unit/timer state, `ip rule show` and
  managed nftables route-policy chains after route-policy apply.
- Route-policy and netpolicy nft comments are rendered as nft string literals.
- Route-policy fails closed instead of marking traffic when no managed backhaul
  candidate is ready.
- Xray/VLESS remote-egress convergence refreshes affected Xray instance
  revisions before route-policy is queued, so `freedom.sendThrough` matches the
  selected live backhaul interface.
- Idempotent backhaul promote/enable acts as a managed repair trigger for
  existing Xray instances with stale remote-egress metadata.
- Operators can queue explicit route-policy cleanup to stop runtime, remove
  managed `fwmark` rules/chains and clean stale destinations from the previous
  node snapshot.

### Firewall

- Default node firewall baseline uses strict input/forward deny, HTTP/HTTPS edge
  allow rules, ICMP/ICMPv6 diagnostics, VPN client forwarding ranges and a
  dedicated `inet megavpn_firewall` table.
- Firewall address groups now have semantic default groups for control-plane,
  operators, VPN client sources, backhaul sources, public service sources and
  blocked destinations.
- Strict firewall preview/apply rejects DNS-only or empty address groups for
  active `accept` matchers because nftables receives only IP, CIDR and range
  entries.
- Strict apply requires a matching successful preview hash for the same policy,
  rules, address groups and safety mode.
- Firewall UI includes posture cards, rule filters, grouped protocol presets and
  explicit apply modes.
- The firewall model diagram connects address groups, rules, policies, apply
  jobs and node apply status as one catalog-to-apply workflow.
- Firewall schema repair covers upgraded installations and address-group
  workflows no longer expose internal identity fields.
- Firewall lifecycle supports typed `node.firewall.disable`, idempotent agent
  removal of the managed firewall table and explicit disabled node state after
  successful completion.
- PostgreSQL migration `000021` aligns `firewall_node_state.status` with the
  disable lifecycle by allowing `pending_disable`, `disabled` and `stale`.
- SSH bootstrap and `Update all agents` refuse to queue when an applied enforced
  node firewall does not contain an active input accept rule for the configured
  SSH port.

### Release Gates And CI

- `make build` covers all production/admin binaries: API, worker, agent,
  migrate and admin.
- CI/self-test/release-gate cover shell syntax across `scripts` and `deploy`,
  frontend JavaScript syntax, docs consistency, action pinning, Go race tests
  and binary `--version` checks.
- Migration drill now checks critical tables, critical indexes, token storage
  invariants, VLESS templates, semantic firewall groups and firewall node-state
  status constraints.
- Privileged job permission logic is consolidated in `internal/app/jobs`, with
  HTTP handlers delegating to the app layer instead of keeping duplicate
  transport-local logic.

### Traffic Accounting

- The summary prioritizes total traffic, received/sent bytes, retained samples,
  clients, nodes, collectors and retention instead of backend prune internals.
- Traffic Accounting uses top-level tabs for Overview, Clients, Collectors,
  Samples and Export.
- Date range, protocol, client, node and row-limit filters are applied by the
  backend to overview summary, recent rows and no-store CSV export through one
  retained-dataset query model.
- Collector status shows node/source/protocol freshness,
  active/degraded/inactive streams, last report time, last bucket and aggregate
  client/sample coverage.
- Expected collector coverage compares active traffic-accounting-enabled Xray,
  WireGuard and OpenVPN runtime revisions with observed sample streams and marks
  missing or partial retained sample streams.
- Retention cleanup is bounded and batched, with PostgreSQL query indexes for
  overview/export filters by bucket, client, node and protocol.

### Client And Instance Lifecycle

- Client VLESS camouflage UX separates public client endpoints from local Xray
  backend endpoints and makes pending provisioning state actionable.
- Hard client deletion cleans PostgreSQL service access, routes, generated
  artifacts, share links, subscriptions, delivery records and service-access
  scoped secrets.
- Instance deletion cascades client service-access cleanup after managed node
  cleanup succeeds.
- Operators can remove a single stale service access from the Client Access
  modal.
- Operator console typography and layout use one UI font stack, safe text
  wrapping and mobile tab grids.

## Validation Gates

The maintained validation entrypoints are:

```bash
scripts/ci/self-test.sh
scripts/ci/release-gate.sh
scripts/smoke/service-pack-staged-smoke.sh --plan <node-id> <endpoint-domain> [certificate-id]
scripts/smoke/service-pack-staged-smoke.sh --cleanup <node-id> <endpoint-domain> [certificate-id]
```

Backward-compatible wrappers remain available at the historical `scripts/*.sh`
paths for existing runbooks and deployments.
