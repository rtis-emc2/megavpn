# External Provider Egress

**Release:** `7.1.1.1`

External egress profiles connect selected client access groups to a commercial
or third-party VPN/proxy provider. They do not replace managed Backhaul and they
do not redirect all traffic generated by a node.

Russian companion: [EXTERNAL_EGRESS_RU.md](EXTERNAL_EGRESS_RU.md).

## Purpose And Boundaries

| Object | Purpose | Traffic scope |
| --- | --- | --- |
| Service instance | Accepts client VPN/proxy traffic | Clients bound to that inbound |
| Managed Backhaul | Carries traffic between MegaVPN ingress and egress nodes | Managed node-to-node paths |
| External egress profile | Connects a runtime node to an external provider | Members of access groups assigned to the profile |

An external profile is global. A deployment materializes that profile on a
concrete runtime node. A client access group references the profile once, while
the profile must be deployed on every node containing an Xray/VLESS instance in
the group scope.

```text
Selected client
  -> global VLESS access group
  -> dedicated Xray outbound with a managed fwmark
  -> dedicated ip rule and routing table
  -> external OpenVPN or WireGuard interface
  -> provider egress

Unselected client or node process
  -> normal Xray policy / main node routing table
```

The node main routing table is not modified. There is no managed
`0.0.0.0/0 -> provider` route in `main`.

## Protocol Status

| Protocol | Profile/import | Node runtime | Activation |
| --- | --- | --- | --- |
| OpenVPN UDP/TCP | `.ovpn`, inline config, separate credentials and PEM material | Ready | Allowed |
| WireGuard | `wg-quick` `.conf`, inline or separate private key | Ready | Allowed |
| SOCKS5 | Structured profile | Preview | Draft only |
| HTTP CONNECT / HTTPS proxy | Structured profile | Preview | Draft only |
| Shadowsocks | URL/JSON/structured catalog entry | Preview | Draft only |
| VLESS | URL/JSON/structured catalog entry | Preview | Draft only |
| L2TP | Structured catalog entry | Planned; intentionally unsafe without IPsec | Draft only |
| L2TP over IPsec | Structured catalog entry | Planned | Draft only |
| IKEv2/IPsec | Structured/mobileconfig catalog entry | Planned | Draft only |
| Trojan | URL/JSON/structured catalog entry | Planned | Draft only |
| Hysteria 2 | URL/YAML/structured catalog entry | Planned | Draft only |

`Preview` and `Planned` mean the data model is reserved and visible, but no
runtime job can activate the protocol. This is a fail-closed compatibility
contract, not a production support claim.

The initial runtime-ready implementation routes IPv4. Provider IPv6 default
routes are not yet materialized.

## Accepted Credential Forms

### OpenVPN

The preferred input is the provider `.ovpn` file. The Control Plane parses and
validates `remote`, `proto`, `dev`, `auth-user-pass` and certificate/key
references.

Supported combinations:

- one inline `.ovpn` containing CA, certificate, private key and credentials;
- `.ovpn` plus separate username and password;
- `.ovpn` plus separate CA certificate, client certificate and private key;
- `.ovpn` plus separate `tls-auth` or `tls-crypt` key.

External filesystem paths, scripts, plugins, management sockets, hooks and
log/status file directives are rejected. The agent removes provider routes,
forces a managed TUN name and adds `route-nopull`, `route-noexec` and
`auth-nocache`.

Binary PKCS#12 upload is not a supported UI workflow yet. Use a provider config
with inline PEM material or upload separate PEM certificate/key material.

### WireGuard

The provider `.conf` can contain the private key, or the private key can be
entered separately. Hook directives such as `PreUp`, `PostUp`, `PreDown` and
`PostDown` are rejected. `SaveConfig` is rejected as well.

Runtime-ready WireGuard external egress requires `AllowedIPs` to contain
`0.0.0.0/0`. MegaVPN replaces `Table` with its dedicated managed table and
ignores provider DNS changes.

### Structured Draft Profiles

For preview/planned protocols, operators can store endpoint, port, transport,
username/password, UUID, PSK and certificate/private-key material as a draft.
Secrets are encrypted, but the profile cannot be applied until a hardened
runtime driver is released.

## Operator Workflow

1. Open `External egress`.
2. Select `New profile`.
3. Choose the provider protocol.
4. For OpenVPN or WireGuard, paste/select the provider config and run
   `Preview import`.
5. Add separate credentials or certificate material when the preview reports a
   required secret.
6. Save the profile as `active`. Non-ready protocols remain `draft`.
7. Select `Deploy` and choose every runtime node hosting an Xray/VLESS instance
   that the target access group can use.
8. Wait for the deployment apply job and run `Probe`.
9. Open `Clients -> Groups`, edit the global VLESS access group and select the
   profile in `External provider gateway`.
10. Preview/sync the group and verify instance apply jobs.

When runtime fields or secrets of an active profile change, all live
deployments move to `pending`. Apply each deployment again before relying on
the updated provider configuration. Probe is structural: it verifies the
managed unit, interface, policy rule and route table; it does not prove that an
arbitrary Internet destination is reachable through the provider.

Do not assign the profile before all scoped runtime nodes have an active
deployment. Group materialization fails closed when a deployment is absent or
unhealthy.

## Routing And Isolation Model

Each deployment receives values unique on its node:

- interface name in the `mgev*` namespace;
- routing table in `40000..48999`;
- firewall mark in `0x4d590000..0x4d59ffff`;
- deterministic, non-overlapping `ip rule` priority.

Allocation is serialized with a PostgreSQL advisory transaction lock and backed
by unique constraints. Xray applies the mark only to the outbound generated for
the selected access group. Cleanup deletes the exact mark/table rule and flushes
only that deployment table.

The dedicated table always keeps a high-metric `unreachable default` while the
group policy is installed. If the provider interface disappears, marked group
traffic fails closed instead of falling through to the node main table. An
explicit deployment cleanup removes the policy rule and its dedicated table.

## Security Model

- Provider configurations and credentials are encrypted in `secret_refs`.
- Read APIs return secret purpose names, never plaintext values.
- Plaintext secrets are injected only into an `apply` job after a concrete node
  claims it and deployment ownership is verified.
- `probe` and `cleanup` jobs never receive secrets.
- Secret replacement deletes the superseded encrypted record transactionally.
- Agent-managed files use mode `0600`; route scripts use `0700`.
- Deployment IDs, interface names, route tables and marks are validated again by
  the agent.
- Apply automatically verifies or installs the required OpenVPN/WireGuard OS
  runtime before stopping an existing deployment.
- API apply/probe/cleanup operations require both node write and access-group
  policy permissions.

## Diagnostics

Control Plane checks:

- profile status is `active` and runtime support is `ready`;
- deployment status is `active`;
- the latest external-egress apply/probe job succeeded;
- group sync and related instance apply jobs succeeded.

Node checks:

```bash
systemctl list-units 'megavpn-external-egress-*' --all --no-pager
systemctl status megavpn-external-egress-<deployment-id-without-dashes>.service --no-pager -l
ip link show
ip rule show
ip route show table <40000-48999>
journalctl -u megavpn-external-egress-<deployment-id-without-dashes>.service -n 200 --no-pager
```

Expected state:

- the managed unit is active;
- the `mgev*` interface exists;
- the dedicated table contains a default route through that interface;
- `ip rule` contains the deployment fwmark and table;
- the main table has not acquired a provider default route.

## Failure And Rollback

| Failure | Behavior | Recovery |
| --- | --- | --- |
| Config parser rejects input | Nothing is deployed | Remove forbidden directives or use separate managed secrets |
| Runtime package install fails | Existing deployment is left running | Fix OS repositories/capabilities and apply again |
| New runtime does not start | Apply job fails and reports stage/output | Inspect unit journal, correct profile, apply again |
| Provider interface goes down | Selected group traffic is rejected by the dedicated unreachable route | Restore the provider runtime or move the group to another active profile |
| Group references missing deployment | Group materialization is blocked | Deploy/apply profile on every scoped node |
| Provider is unavailable | Probe/deployment becomes failed or degraded | Restore provider service or move group to another active profile |

To roll back group traffic, edit the access group and clear `External provider
gateway`, then sync/apply. To remove runtime state, run deployment `Cleanup`.
Delete the profile only after all group references are cleared and all
deployments are inactive.

The enforced removal order is: unassign the profile from every group, sync the
affected instances, clean up every node deployment, disable the profile, then
delete it. PostgreSQL lifecycle guards enforce the same invariants as the API,
including under concurrent operator requests.

## Observability, Backup And Scaling

Profile/deployment lifecycle and jobs are audit-visible. Provider tunnel unit,
interface, route and rule health are returned by probe jobs. Existing client
traffic accounting remains associated with client/service access; provider
credentials and packet payloads are not logged.

Database backup contains encrypted profile secrets. Restore also requires the
same external secret master key. Node runtime files are materialized state and
must be recreated from the Control Plane after disaster recovery.

One profile can serve many access groups. Job count scales with deployments and
affected Xray instances, not with individual group members. Capacity planning
must still account for provider tunnel throughput, node CPU and Xray outbound
concurrency.
